# AETHERIUM-GENESIS: รายงานการตรวจสอบโครงสร้างระบบโดยละเอียด

## 1. ภาพรวมของระบบ (System Overview)
**AETHERIUM-GENESIS** เป็นระบบ "เครื่องยนต์แห่งเจตจำนง" (Intent Engine) ที่ขับเคลื่อนด้วย AI เพื่อสร้าง "อินเทอร์เฟซที่มีชีวิต" (Living Interface) สถาปัตยกรรมถูกสร้างขึ้นบนแบบจำลองการรับรู้สองเส้นทาง:
- **Javana (Reflex):** จัดการการตอบสนองทางประสาทสัมผัสทันทีและการโต้ตอบที่รวดเร็ว
- **Logenesis (Deep):** จัดการการตีความเจตจำนงที่ซับซ้อน วิวัฒนาการของสถานะ และการคำนวณ "Drift" โดยใช้โมเดลเช่น Gemini

ระบบเน้นปรัชญา "แสงคือเจตจำนง" (Light as Intent) ซึ่ง UI ประกอบด้วยอนุภาคที่ลื่นไหลและการซ้อนทับแบบบรรยากาศ แทนที่จะเป็นปุ่มและเมนูเชิงกลแบบดั้งเดิม

## 2. รายละเอียดโครงสร้างทั้งหมด (Full Structural Breakdown)

### 2.1 ซอร์สโค้ดหลัก (`src/`)
- **`src/backend/`**: ระบบประสาทของแอปพลิเคชัน
    - **`genesis_core/`**: ไดเรกทอรีที่สำคัญที่สุด ประกอบด้วย "DNA ดิจิทัล", การจัดการหน่วยความจำ (Akashic) และเครื่องยนต์ Logenesis
    - **`core/`**: การกำหนดค่าทั่วไปและตรรกะเครื่องยนต์รุ่นเก่า
    - **`departments/`**: ความพยายามในการออกแบบที่เน้นโดเมน (Domain-Driven Design) ปัจจุบันมีโมดูลสำหรับ Design, Marketing, Presentation และ Development
        - *หมายเหตุ: หลายแผนกย่อย (Accounting, Decision) เป็นเพียงโฟลเดอร์ว่าง*
    - **`routers/`**: จุดเชื่อมต่อ FastAPI สำหรับ Aetherium, Entropy และ Metrics
    - **`security/`**: จัดการรหัสผ่านและการเข้าถึง
    - **`worker_drones/`**: โครงสร้างพื้นฐานสำหรับงานเบื้องหลัง (ปัจจุบันยังมีน้อย)
- **`src/frontend/`**: "ภาชนะ" (Vessel) หรือร่างกายส่วนหน้าของระบบ
    - **`public/`**: ทรัพยากร, ไอคอน และ Service Workers
    - **`public/gunui/`**: "Legacy Actuator UI" ที่ทำงานซ้ำซ้อนกับอินเทอร์เฟซหลัก

### 2.2 เอกสารและข้อมูลเมตา (`docs/`, root)
- **`docs/`**: พิมพ์เขียว, ประกาศเจตจำนง และข้อมูลจำเพาะทางเทคนิคอย่างละเอียดทั้งภาษาอังกฤษและภาษาไทย
- **`data/`**: การจัดเก็บข้อมูล JSON แบบถาวรสำหรับ "Akashic Records" (หน่วยความจำหลัก)
- **ไฟล์ใน Root**:
    - `awaken.py`: จุดเริ่มต้นหลักสำหรับการเปิดใช้งานระบบ
    - `ARCHITECTURE.md`: แผนที่ทางเทคนิคระดับสูง
    - `GENESIS_OMNI_STATE_MANIFEST.md`: สถานะปัจจุบันของการรับรู้ระบบ

### 2.3 ระบบเก่าและส่วนจัดเก็บ (`legacy/`)
- ไดเรกทอรีนี้ประกอบด้วยการพัฒนาในอดีตหลายรุ่น:
    - `legacy_v1/`: เวอร์ชันก่อนหน้าขนาดใหญ่ รวมถึงสเปค Kivy และโมดูลย่อยต่างๆ (niyama, inspira)
    - `gun_ui_integration/`: โปรโตไทป์แรกของ Gun UI
    - `ai_utils_package/`: ฟังก์ชันยูทิลิตี้ทั่วไปที่ปัจจุบันถูกรวมเข้ากับ `src` บางส่วนแล้ว

### 2.4 การทดสอบ (`tests/`)
- ชุดการทดสอบที่ครอบคลุมทั้ง API, ความปลอดภัย, ฟิสิกส์ (LCL) และลูป Logenesis รวมถึงการทดสอบประสิทธิภาพแบบแมนนวลสำหรับส่วนประกอบประสิทธิภาพสูง

## 3. รายละเอียดโครงสร้างที่ไม่สอดคล้องและส่วนเกิน (Inconsistencies & Redundancies)

### 3.1 อินเทอร์เฟซผู้ใช้แบบแยกส่วน (Split-Brain UI)
- **ปัญหา**: มีจุดเข้าใช้งานหน้าบ้านสองจุดที่แตกต่างกัน: `index.html` ใน root และ `src/frontend/public/gunui/index.html`
- **ผลกระทบ**: ทั้งสองจัดการระบบอนุภาคและเสียง/เจตจำนง แต่ใช้โปรโตคอลและการจัดสไตล์ที่ต่างกัน สร้างประสบการณ์ผู้ใช้ที่แตกแยกและเพิ่มภาระในการดูแลรักษาเป็นสองเท่า

### 3.2 การทำซ้ำของโปรโตคอล WebSocket
- **ปัญหา**: `src/backend/main.py` รักษาจุดเชื่อมต่อ WebSocket สองจุด: `/ws` (เลิกใช้งาน/รุ่นเก่า) และ `/v2/ws` (ใช้งานอยู่)
- **ผลกระทบ**: จุดเชื่อมต่อรุ่นเก่ายังถูกใช้งานโดย "Actuator UI" ทำให้ไม่สามารถเปลี่ยนไปใช้โปรโตคอล V2 ที่มีประสิทธิภาพมากกว่าได้อย่างสมบูรณ์โดยไม่ทำให้ส่วนประกอบเสียหาย

### 3.3 การพึ่งพาที่ถูกทิ้งไว้ / หนักเกินไป (Orphaned / Heavy Dependencies)
- **ปัญหา**: `src/backend/private/advanced_diffusion.py` นำเข้า `torch` และ `diffusers`
- **ผลกระทบ**: สิ่งเหล่านี้เป็นไลบรารีขนาดใหญ่ (หลาย GB) ที่ปัจจุบัน**ไม่ได้ถูกรวมเข้ากับ**ขั้นตอน Logenesis ที่ใช้งานอยู่ ทำให้ระบบบวมโดยไม่จำเป็นสำหรับฟีเจอร์ที่ยังไม่ได้เปิดใช้งาน

### 3.4 แผนกโดเมนที่ว่างเปล่า (Empty Domain Stubs)
- **ปัญหา**: `src/backend/departments/` ประกอบด้วยโฟลเดอร์ว่างหลายโฟลเดอร์ (เช่น `accounting`, `decision`)
- **ผลกระทบ**: สร้าง "เสียงรบกวนในไดเรกทอรี" และทำให้โค้ดดูเหมือนยังไม่เสร็จหรือมีการออกแบบที่ซับซ้อนเกินความจำเป็น

## 5. รายละเอียดข้อเสนอแนะ (Detailed Recommendations)

1. **การรวมโปรโตคอล (Protocol Unification)**: สร้างโปรโตคอล `AetheriumStandard` เดียวสำหรับ WebSockets ที่รองรับทั้ง "Reflex" (รวดเร็ว) และ "Logenesis" (ลึกซึ้ง) ในสตรีมเดียว
2. **การปรับโครงสร้าง UI เน้นแสง (Light-First UI Refactor)**:
    - นำแท็ก `<input>` มาตรฐานของ HTML ออก
    - ใช้ระบบอนุภาคที่มีอยู่สร้าง "Visual Echoes" (เงาสะท้อนทางสายตา) ของสิ่งที่ผู้ใช้พิมพ์
    - เปลี่ยนปุ่ม "Recall" เป็นท่าทาง "การดูดซับอนุภาค" (Particle Absorption)
3. **การตัดการพึ่งพา (Dependency Pruning)**: ตรวจสอบ `requirements.txt` และนำ `torch` และ `diffusers` ออกจากสภาพแวดล้อมหลักเพื่อให้ระบบสามารถทำงานบนอุปกรณ์ขนาดเล็กได้
4. **การกำหนดมาตรฐานชื่อ (Namespace Standardization)**: เปลี่ยนชื่อโฟลเดอร์ทั้งหมดใน `src` ให้เป็นรูปแบบ `snake_case` ที่เคร่งครัด เพื่อลดความสับสนของชื่อที่ซ้ำกัน (เช่น `data` ที่อยู่ในหลายที่)

## 6. ส่วนเกินที่ไม่สามารถจัดประเภทได้ (Unclassifiable Extra Items)
- **`awaken.py` เทียบกับ `main.py`**: ยังไม่ชัดเจนว่าไฟล์ใดเป็นตัวรัน "การใช้งานจริง" (Production) หลัก `awaken.py` ดูเน้นไปทางปรัชญา ในขณะที่ `main.py` เป็นเซิร์ฟเวอร์ FastAPI ที่ใช้งานได้จริง
- **`access_keys.json`**: ข้อมูลนี้ควรอยู่ในไฟล์ `.env` หรือระบบจัดการความลับ ไม่ควรเป็นไฟล์ข้อความธรรมดาใน root
